# Memo App MVP Roadmap

This roadmap tracks the path from the current KotoShell/KotoSim foundation to a
working memo app that can be launched, edited, saved, reloaded, and verified in
KotoSim before hardware bring-up.

## Current State

- KotoShell can list mock packages from `sdcard_mock/apps` and request launches.
- KotoSim can render the shell to a framebuffer, BMP, or optional live window.
- KotoIME core pieces exist for romaji/kana, Sticky Shift, text layout, and SKK
  dictionary indexing.
- KotoFS sandbox path resolution and host filesystem adapters exist.
- The `KBC1` runtime ABI is documented, and the first bytecode verifier exists
  in `koto-core::runtime`.
- The bytecode app development environment is now tracked separately in
  [BYTECODE_APP_DEV_ROADMAP.md](BYTECODE_APP_DEV_ROADMAP.md), including the
  high-level app language, compiler, SDK prelude, and app build loop.

## Scope

This roadmap is scoped to simulator-verifiable behavior for both the memo app
and IME. The target is not only that `dev.koto.memo` launches, but that KotoSim
can prove the full editing loop:

- launch the memo app from KotoShell;
- enter ASCII text and Japanese text through KotoIME;
- show romaji/kana composition and SKK-style conversion state in the fixed IME
  line;
- commit converted text into the memo document;
- save through sandboxed file APIs;
- exit, relaunch, and reload the saved content.

Out of scope for this roadmap: real PicoCalc keyboard-matrix confirmation,
embedded FAT/SD timing, LCD transfer performance, and hardware power-loss save
behavior. Those become device validation issues after the embedded HAL can run
the same simulator scenarios.

## Target Outcome

The memo app MVP is complete when KotoSim can launch `dev.koto.memo`, accept
scripted and interactive text input, verify IME composition/conversion behavior,
show composition in the fixed IME line, save memo text through sandboxed file
APIs, reload the saved memo, and pass the standard local checks.

## Working Order

Treat KOTO-0033 through KOTO-0035 as the runtime spine. Once those pass, the
memo-specific work should proceed in pairs: host calls plus editor model first,
then IME integration plus packaged fixture, then end-to-end validation.

Hardware work is intentionally out of this roadmap. The memo app should prove
the portable core and host simulator path first; real PicoCalc validation can
reuse the same scenarios after the embedded HAL is ready.

The memo app should be source-authored in the high-level app language once that
toolchain exists. Hand-authored assembly is acceptable for VM and ABI fixtures,
but not for the long-lived interactive memo implementation.
