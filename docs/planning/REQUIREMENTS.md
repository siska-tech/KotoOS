## 1. はじめに

### 1.1 本書の目的

本書は、ClockworkPi 社製の MCU 搭載携帯端末「PicoCalc」上で動作する軽量日本語 PDA／アプリケーション実行環境「**KotoOS**」の要件を定義する。[Research.md](Research.md) のハードウェア調査および実現性評価の結論を前提とし、開発対象範囲・機能要件・非機能要件・制約・段階的開発計画を明確化することを目的とする。

### 1.2 プロジェクトのコンセプト

KotoOS は、携帯ゲーム機「P/ECE（ピース）」風の小型アプリ実行環境と、BTRON/TiPO 系の思想を受け継いだ軽量日本語 PDA 環境を統合する。限られたリソース（RP2040: 264KB SRAM）の中で、フルカラー 320×320 画面でのビジュアルノベルや DOS 風ミニゲーム、SKK 風の軽量日本語入力を実現し、「ハードウェアを直接叩き切るハッカーの遊技場」としてのレトロ／モダンな OS 空間を構築する。

### 1.3 用語定義

| 用語            | 定義                                                       |
| :-------------- | :--------------------------------------------------------- |
| メイン MCU      | RP2040 / RP2350。OS 本体が動作するコアプロセッサ           |
| サウスブリッジ  | STM32 MCU。キーボードスキャン・電源／バッテリー管理を担当  |
| HAL             | Hardware Abstraction Layer。ハード依存コードを隔離する層   |
| PSRAM           | 基板搭載の 8MB 外部メモリ（PIO 駆動・SPI 接続）            |
| .kpa            | Koto Package Archive。アプリ／アセットを単一化した配布形式 |
| 差分描画        | Dirty Rectangle。変更矩形のみを LCD へ転送する描画手法     |
| Sticky Shift    | 一度押すと次の 1 ストロークのみ大文字化する入力補助機能    |
| バイトコード VM | ネイティブ実行が困難なため採用するアプリ実行用仮想マシン   |
| Rust            | KotoOS 本体・HAL・ツール群の主要実装言語                   |

## 2. 対象ハードウェアと前提制約

KotoOS の設計は以下のハードウェア仕様および制約を絶対的な前提とする（詳細は [Research.md](Research.md) 第 1〜6 章参照）。

### 2.1 ハードウェア仕様サマリ

| 項目       | 仕様                                                                                            |
| :--------- | :---------------------------------------------------------------------------------------------- |
| メイン MCU | RP2040（Cortex-M0+ 133MHz / 264KB RAM）。差し替えで RP2350（Cortex-M33 150MHz / 520KB RAM）対応 |
| Flash      | 2MB〜16MB（標準キットは 2MB）                                                                   |
| PSRAM      | 8MB（SPI 接続、RP2040 では PIO 経由・XIP 不可）                                                 |
| 画面       | 4 インチ IPS LCD、320×320、ILI9488 / ST7365P 互換、SPI 接続                                     |
| キーボード | 67 キー QWERTY、STM32 経由、I2C（0x1F）でポーリング                                             |
| 音声       | デュアル PWM スピーカー（GP26/GP27 が同一スライス共有）                                         |
| ストレージ | MicroSD（SPI0 接続、FAT ファイルシステム）                                                      |
| 電源       | 18650 リチウムイオン電池、USB Type-C 充電、I2C で残量取得                                       |

### 2.2 設計上クリティカルな制約（前提）

| ID   | 制約                                                                          | 影響                                                    |
| :--- | :---------------------------------------------------------------------------- | :------------------------------------------------------ |
| HC-1 | SRAM が 264KB しかなく、フルスクリーンフレームバッファ（200KB）を常駐できない | 差分描画／ラインバッファ方式が必須                      |
| HC-2 | SPI 描画帯域の限界により全画面更新は実効 15〜16fps が上限                     | 全画面更新を避ける描画戦略が必須                        |
| HC-3 | PSRAM は CPU メモリ空間に直接マッピング不可（XIP/実行不可）                   | ネイティブバイナリ動的実行が不可能。VM 方式が必須       |
| HC-4 | GP26/GP27 が同一 PWM スライスを共有                                           | ハードウェアステレオ和音不可。ソフト PCM ミキサーが必須 |
| HC-5 | I2C キーボードがファーム次第で 10kHz に制限され遅延発生                       | 起動時 100kHz 以上への初期化が必須                      |
| HC-6 | SD カードは SPI モードのみ。ランダムアクセスが遅い                            | シーケンシャルリード／パッケージ化が必須                |
| HC-7 | 主要 GPIO がハードワイヤード割当                                              | ピンアサイン順守、外部拡張余地は小                      |
| HC-8 | キーマトリクス構造上、同時押し時にゴースト／ブロッキングが発生しうる          | 標準ゲーム入力マッピングの検証が必須                    |

### 2.3 固定 GPIO ピンアサイン

PicoCalc の主要ペリフェラルは以下の GPIO に固定されている。実機 HAL はこの割当を前提とし、任意のピン再配置を要件としない。

| 用途         | バス／信号   | GPIO                                    |
| :----------- | :----------- | :-------------------------------------- |
| LCD          | SPI1         | SCK=GP10, MOSI=GP11, MISO=GP12, CS=GP13 |
| LCD 制御     | DC / RESET   | DC=GP14, RESET=GP15                     |
| MicroSD      | SPI0         | MISO=GP16, CS=GP17, SCK=GP18, MOSI=GP19 |
| MicroSD 検出 | DETECT       | GP22                                    |
| PSRAM        | PIO SPI 相当 | CS=GP20, SCK=GP21, MOSI=GP2, MISO=GP3   |
| STM32        | I2C1         | SDA=GP6, SCL=GP7                        |
| 音声         | PWM          | 左=GP26, 右=GP27                        |

## 3. スコープ

### 3.1 対象範囲（In Scope）

- KotoOS カーネル／HAL 層（Rust によるハード抽象化）
- 画面・入力・音声・ファイルの共通 API（KotoSDK）
- 日本語入力（KotoIME）・フォント描画（KotoFont）
- ランチャー（KotoShell）・ファイル管理（KotoFS）
- アプリ実行環境（KotoRuntime：バイトコード VM）
- アプリケーションエンジン群（KotoVN / KotoDOS / KotoMML / PicoMings）
- PC 上の開発用シミュレータ（KotoSim：SDL2 バックエンド）
- 将来拡張としての対話型スクリプト環境（KotoREPL）

### 3.2 対象外（Out of Scope）

- マルチユーザー機能・ネットワーク通信（Wi-Fi/Bluetooth を用いた通信スタック）
- 連文節変換 IME（メモリ制約上、SKK 風方式に限定）
- ネイティブバイナリの動的ロード実行（HC-3 により不可）
- 完全な Linux 互換環境・POSIX プロセスモデル

### 3.3 初期ターゲットと将来拡張

- **初期ターゲット**: 最も制約の厳しい RP2040（264KB SRAM）上での安定動作を必須とする。
- **将来拡張**: RP2350 / Pico Plus 2W 等への換装時に QSPI PSRAM の直接マッピングや性能向上を活かせる設計余地を残す（必須要件ではない）。

## 4. 機能要件

各機能モジュールの要件を定義する。優先度は MoSCoW（Must / Should / Could）で示す。

### 4.1 KotoShell（ランチャー）— 優先度: Must

- FR-SHELL-1: SD カード上の `.kpa` パッケージ一覧を KotoFS 経由で取得・表示すること。
- FR-SHELL-2: 十字キーと Enter で項目選択・アプリ起動ができる軽量 UI を提供すること。
- FR-SHELL-3: UI 更新は差分描画を用い、カーソル移動・選択を 30fps 以上で滑らかに行うこと。
- FR-SHELL-4: Rust ネイティブコードとして OS 本体に組み込むこと。
- FR-SHELL-5: バッテリー残量を取得できる場合、ステータス領域に残量または低電圧警告を表示すること。

### 4.2 KotoSDK（システム API / HAL）— 優先度: Must

- FR-SDK-1: 画面描画 API をコールバックによるライン更新インターフェースとして提供すること。
- FR-SDK-2: 入力を構造体ポーリング（上下左右、A/B/X/Y 相当）で取得できること。
- FR-SDK-3: 音声をオーディオバッファ要求インターフェースとして提供すること。
- FR-SDK-4: ファイル I/O を標準ストリーム類似の API で提供すること。
- FR-SDK-5: 同一 API が KotoSim（PC）と実機の双方で動作すること（HAL 分離）。
- FR-SDK-6: 標準ゲームパッドマッピングは、PicoCalc のキーマトリクスにおける同時押し・ゴースト・ブロッキングを検証したキー組み合わせを既定値とすること。
- FR-SDK-7: STM32 サウスブリッジから取得可能な電源／バッテリー情報を、HAL 経由の任意機能 API として公開すること。

### 4.3 KotoRuntime（アプリ実行環境）— 優先度: Must

- FR-RT-1: 軽量バイトコード VM（Wasm3 / mruby / Lua / 独自スタックマシンのいずれか）を OS に組み込むこと。
- FR-RT-2: PSRAM 上のバイトコードをシリアルに読み出しながら解釈・実行すること（ネイティブ実行は行わない）。
- FR-RT-3: VM の内部ステートが消費する SRAM を数十 KB 程度に抑えること。
- FR-RT-4: アプリを Sandbox 環境で実行し、OS クラッシュを防止すること。
- FR-RT-5: PSRAM 上のバイトコード／アセットは直接ポインタ参照せず、PSRAM ブロック転送 API 経由で SRAM バッファに読み出して扱うこと。

### 4.4 KotoIME（日本語入力）— 優先度: Must

- FR-IME-1: ローマ字かな入力＋ SKK 風（大文字始まりを変換トリガとする）変換を提供すること。
- FR-IME-2: Sticky Shift を実装し、親指タイピングで逐次 1 ストロークの大文字入力ができること。
- FR-IME-3: 画面下部に 1〜2 行の専用 IME 入力ラインを常時確保すること。
- FR-IME-4: SKK 辞書は SD カード上に配置し、行頭文字オフセットのインデックステーブルを SRAM 常駐させてバイナリサーチで検索すること。

### 4.5 KotoFont（日本語フォント描画）— 優先度: Must

- FR-FONT-1: 「美咲フォント」「k8x12」等の軽量ビットマップフォントを採用すること。
- FR-FONT-2: JIS 第 1・第 2 水準を網羅する 16×16 フォント（約 300KB）等を PSRAM に常駐させること。
- FR-FONT-3: 描画要求時に該当グリフのみを PSRAM→SRAM へ読み出してラスタライズすること。

### 4.6 KotoFS（ファイル管理）— 優先度: Must

- FR-FS-1: FatFs のラッパーとして FAT16/FAT32 SD カードへアクセスすること。
- FR-FS-2: アプリからは自身の Sandbox ディレクトリ配下のみアクセス可能な仮想パス機構を提供すること。
- FR-FS-3: アセットはシーケンシャルリードを徹底し、起動時に必要ブロックを一括事前ロードすること。

### 4.7 KotoVN（ビジュアルノベルエンジン）— 優先度: Should

- FR-VN-1: KotoRuntime 上で動作する専用スクリプトパーサを提供すること。
- FR-VN-2: 背景画像は 256 色インデックスカラー＋ RLE 等で圧縮し、SD→PSRAM 展開後に必要矩形のみを描画すること。

### 4.8 KotoDOS（DOS 風ゲーム）— 優先度: Should

- FR-DOS-1: 画面上部 320×200 領域のみを更新ターゲットとし、下部 320×120 を静的 UI として保持すること。
- FR-DOS-2: 320×200 領域で 25〜30fps を達成すること。

### 4.9 KotoMML（音楽／効果音環境）— 優先度: Should

- FR-MML-1: ソフトウェア PCM ミキサー方式により、複数トラック MML 再生とポリフォニックな効果音を実現すること。
- FR-MML-2: PWM キャリア周波数を可聴域外（64〜128kHz）に固定し、タイマー割り込みで波形合成すること。
- FR-MML-3: BGM と効果音の同時発音を可能とすること。

### 4.10 PicoMings（レミングス風ゲーム）— 優先度: Could

- FR-PM-1: 多数スプライトと背景タイルマップをスキャンライン単位で合成する軽量 2D スプライトエンジンを提供すること。
- FR-PM-2: 方向キーと Enter/Space のみで操作が完結すること。

### 4.11 KotoSim（PC シミュレータ）— 優先度: Must

- FR-SIM-1: SDL2 で 320×320 の仮想ウィンドウを生成し、実機到着前の開発を可能とすること。
- FR-SIM-2: PC キーボードイベントを PicoCalc キーマトリクス構造体へマッピングすること。
- FR-SIM-3: PC ローカルディレクトリ（例: `./sdcard_mock/`）をルート FS としてマウントすること。
- FR-SIM-4: KotoOS の本体ソースコードを実機と共通化（HAL のみ差し替え）すること。
- FR-SIM-5: PC 版は Rust の通常ツールチェーン（`cargo run` / `cargo test`）でビルド・検証できること。

### 4.12 KotoREPL（対話型スクリプト環境）— 優先度: Could

- FR-REPL-1: KotoShell から軽量スクリプトまたは VM バイトコードの対話的な実行環境を起動できる設計余地を残すこと。
- FR-REPL-2: KotoREPL は MVP の必須機能とはせず、KotoRuntime 完成後の拡張機能として扱うこと。

## 5. 非機能要件

### 5.1 性能要件

| ID         | 要件                                                                                                   |
| :--------- | :----------------------------------------------------------------------------------------------------- |
| NFR-PERF-1 | KotoShell 等の GUI は差分描画により 30fps 以上の UI 応答を確保すること                                 |
| NFR-PERF-2 | 全画面更新が必要な場面は 15fps を許容範囲とし、極力 160×160 仮想画面や RGB111 8 色モードを活用すること |
| NFR-PERF-3 | KotoDOS の 320×200 ゲーム領域は 25〜30fps を達成すること                                               |
| NFR-PERF-4 | キー入力はゲームループ（約 16.6ms 周期）内でノンブロッキングにポーリングし、レイテンシを最小化すること |
| NFR-PERF-5 | I2C バスクロックを起動時に最低 100kHz（可能なら 400kHz）に初期化すること                               |
| NFR-PERF-6 | ゲーム用途の標準キー割当は、方向キー＋アクションキーの同時押し成立性を実機で検証すること               |

### 5.2 メモリ要件

| ID        | 要件                                                                                                 |
| :-------- | :--------------------------------------------------------------------------------------------------- |
| NFR-MEM-1 | OS コア常駐分（カーネル・FatFs バッファ・USB スタック・スタック・ヒープ）を約 80〜100KB に抑えること |
| NFR-MEM-2 | アプリ・フレームバッファ・フォントキャッシュは残り約 150〜180KB の範囲でやり繰りすること             |
| NFR-MEM-3 | フルスクリーンフレームバッファ（200KB）を SRAM に常駐させないこと                                    |
| NFR-MEM-4 | PSRAM は「高速スワップ／キャッシュストレージ」として API 経由でブロック転送運用すること              |
| NFR-MEM-5 | RP2040 版では PSRAM のメモリマップ／XIP／直接ポインタ参照に依存する実装を禁止すること                |

### 5.3 描画アーキテクチャ要件

- NFR-DRAW-1: 用途に応じ、差分描画（Dirty Rectangle）とラインバッファ／ダブルバッファ方式を動的に切り替えること。
- NFR-DRAW-2: DMA を用いて CPU 非介在のノンブロッキング SPI 転送を行うこと。
- NFR-DRAW-3: 画面モード（320×320 フルカラー／160×160 仮想／320×200 DOS 風／RGB111 8 色）をサポートすること。

### 5.4 移植性・保守性要件

- NFR-PORT-1: コア OS ロジック（UI・FS・VM・文字描画）は可能な限り `no_std` 互換な Rust で記述し、ハード依存を一切含めないこと。
- NFR-PORT-2: ハード依存は HAL（`hal_video_update` / `hal_input_poll` / `hal_audio_callback` / `hal_fs_read` 等）に完全隔離すること。
- NFR-PORT-3: PC バックエンドと PicoCalc 実機バックエンドの 2 系統の HAL 実装を保持すること。
- NFR-PORT-4: 実機 HAL は第 2.3 節の固定 GPIO ピンアサインに従い、ピン割当差分をコア OS 側へ漏らさないこと。
- NFR-PORT-5: C/C++ ライブラリ（FatFs、既存 PSRAM ドライバ、VM 等）を利用する場合は Rust FFI 境界を HAL または外部アダプタ層に閉じ込めること。

### 5.5 信頼性・安全性要件

- NFR-REL-1: アプリは Sandbox（仮想パス＋ VM）内で実行し、OS 本体・他アプリのメモリ／ファイルを破壊しないこと。
- NFR-REL-2: 低品質・大容量 SD カードでの初期化失敗に備え、SPI クロックのフォールバック処理を行うこと。
- NFR-REL-3: オーディオはリングバッファ＋ DMA により音切れを防止すること。
- NFR-REL-4: バッテリー残量取得 API が利用可能な場合、低残量時にユーザーへ警告し、書き込み中のデータ破損を避ける設計とすること。

### 5.6 開発・運用要件

- NFR-DEV-1: 実機への書き込みは UF2 ファイルを BOOTSEL モードでドラッグ＆ドロップする方式とすること。
- NFR-DEV-2: デバッグログは USB-CDC（仮想シリアル）へリダイレクトし、PC ターミナルで監視できること。
- NFR-DEV-3: 開発の大部分（目標 80%）を実機到着前に KotoSim 上で完結できること。
- NFR-DEV-4: Rust 標準の `cargo fmt` / `cargo clippy` / `cargo test` を PC 側の基本検証コマンドとして採用すること。

---

## 6. アプリケーション配布要件

- FR-PKG-1: アプリ（P/ECE 風アプリ・ビジュアルノベル等）は単一の `.kpa`（Koto Package Archive）形式に統合して配布・実行すること。
- FR-PKG-2: アセット（画像・MML・フォント）はパッケージ内で連続セクタに配置し、シーケンシャルリードを保証すること。
- FR-PKG-3: アプリ開発者は PC 上で Rust／スクリプト等を記述しバイトコードへコンパイルして `.kpa` にパッキングできること。

| KotoSim     | 極めて高い | 低     | 特になし                                                 |

## 8. 開発ロードマップ（MVP）

[Research.md](Research.md) 第 9.3 章のフェーズ計画を要件として採用する。

### フェーズ 1: KotoSim 構築とコア API 確立（PC 環境のみ）

- SDL2 シミュレータ環境のセットアップ。
- 画面描画 API・フォントラスタライズ（美咲フォント表示）・キー入力ポーリングの Rust trait / API 定義。
- モック FS を用いた KotoShell プロトタイプ作成と画面遷移テスト。

### フェーズ 2: 実機移植とベアメタル駆動

- Rust 組み込み開発環境（RP2040/RP2350 ターゲット、probe-rs 等）を構築し、フェーズ 1 コードを実機向けにクロスコンパイル。
- 実機 HAL 実装。まず低速でも ILI9488 への SPI 描画を成立させる。
- STM32 からの I2C キーボード入力取得（10kHz バグ回避とレイテンシ検証）。

### フェーズ 3: 最適化と KotoOS 基盤完成

- 描画ルーチンを DMA ＋ラインバッファ方式へ書き換え、FPS 向上。
- FatFs を組み込み SD カードからのリソース読み込みを実装。
- SKK 風 IME（Sticky Shift）を実装し、メモ帳アプリを完成。

### フェーズ 4: アプリエンジンと音響統合

- ソフトウェア PCM ミキサーを PWM 割り込みへ統合し効果音を発音。
- バイトコード VM を組み込み KotoRuntime を完成。
- SD カード上の KotoDOS 風ミニゲーム・KotoVN がランチャーから起動・実行できることを実証。

## 9. 制約・前提・リスク

### 9.1 技術的制約

第 2.2 節の HC-1〜HC-8 を必須前提とする。とりわけ HC-3（PSRAM 実行不可）は KotoRuntime を VM 方式に決定づける根本制約である。

### 9.2 主要リスクと対策

| リスク                              | 影響             | 対策                                              |
| :---------------------------------- | :--------------- | :------------------------------------------------ |
| VM のフットプリント／速度が要件未達 | KotoRuntime 全体 | 早期に複数 VM 候補（Wasm3/mruby/独自）を PoC 比較 |
| SD カード相性問題                   | 起動・データ読込 | SPI クロックフォールバック、推奨 SD カードの明示  |
| ソフト PCM ミキサーの CPU 負荷過大  | 音声＋描画の両立 | Core1 専用割当・サンプリングレート調整            |
| I2C キーボード遅延                  | 操作性全般       | 100/400kHz 初期化、ノンブロッキングポーリング     |
| キー同時押しのゴースト／ブロック    | ゲーム操作性     | 実機検証済みの標準ゲームパッド割当を採用          |
| バッテリー残量取得の不安定性        | 携帯利用・保存   | 取得不能時の表示フォールバック、低残量時の警告    |
| Rust 組み込み HAL の成熟度差         | 実機 HAL         | 早期 PoC、必要に応じた C ライブラリ FFI 隔離      |
| SRAM 枯渇                           | OS 全体の安定性  | 画面モード切替・PSRAM スワップ活用の徹底          |

### 9.3 前提

- 標準キット（Pico 1H / RP2040 / 2MB Flash）を初期ターゲットとする。
- 開発言語は Rust を基本とし、C/C++ は既存ドライバやライブラリ連携が必要な場合の FFI 対象に限定する。
- 実機到着前の開発は KotoSim（SDL2）で先行する。

## 10. 参考・継承元

KotoOS は以下の歴史的・現代的環境の思想を継承する（[Research.md](Research.md) 第 10 章）。

- **P/ECE**: 精神的後継。256KB 級 SRAM での緻密なメモリ管理哲学を継承。
- **BrainPad TiPO / BTRON**: 画面下部固定 IME 入力ライン、データカプセル化のファイル管理思想を KotoIME / KotoFS に採用。
- **DOS/VGA 小型ゲーム**: 320×200・256 色の描画アルゴリズムを KotoDOS で再現。
- **ポケコン（PC-G850 等）**: シェルからの REPL 的スクリプト実行を志向。

既存移植事例（Fuzix OS / uMac / PicoMite 等）を技術参照として最大限活用する。
