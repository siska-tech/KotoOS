# KotoUI Input and Focus Contract

KOTO-0209 adds device-independent input pulses and a fixed-capacity focus
registry. Physical scan codes and host window keys remain in their existing HAL
or simulator layers.

## Event normalization

`UiEvent` combines a semantic `UiAction` with `EventPhase::Pressed`,
`EventPhase::Repeated`, or `EventPhase::Released`. Actions cover four-way navigation, explicit next/
previous traversal, activation, cancellation, Unicode text, backspace, delete,
home, end, and submit.

KotoCore provides adapters for both `InputState` and `VmInputSnapshot`. Existing
snapshots represent initial and repeated keys as pulses in the same pressed or
intent field, so an adapter caller supplies the subset known to be generated by
repeat. This retains repeat behavior without adding device codes to KotoUI or
changing the VM host ABI.

When several inputs occur in one VM frame, the adapter emits them in this order:
up, down, left, right, backspace, delete, home, end, submit, cancel, then the
typed Unicode scalar. Native `InputState` emits directions, activate, cancel,
then Unicode text; a release follows that button's pressed pulse when both are
present in one snapshot. Invalid Unicode and IME-owned shift/convert/commit intents do
not become component events.

Adapters write to caller-owned `EventBuffer<N>`. On capacity exhaustion they
return `EventBufferFull`; no allocation, I/O, blocking, or implicit event drop
occurs.

## Focus registry

`FocusManager<N>` defaults to 16 registered entries. Each entry contains a
stable `WidgetId`, absolute focus-indicator bounds, a caller-assigned scope, and
enabled/visible flags. Registration order is traversal order.

- Focus is explicit through `focus`, `focus_first`, `move_next`, or
  `move_previous`; registering a component does not silently focus it.
- Disabled, hidden, empty-bounds, and wrong-scope entries are skipped.
- Traversal wraps within the active scope.
- Removing or disabling the focused entry selects the next eligible entry at
  its registration position, wrapping to the first eligible entry.
- Focus changes damage only the old and new indicator bounds.
- Directional events are first dispatched to the focused component. A caller
  invokes traversal only if the component does not consume that direction, so a
  list or text field can keep its own arrow-key behavior.

## Modal scope

Scope zero is the root. One non-root modal scope can be active at a time. Opening
it records the prior scope/focus, traps traversal to eligible entries in the
modal scope, and uses the requested initial focus or its first eligible entry.
Closing or cancelling restores the prior focus when it is still eligible;
otherwise it selects the first eligible entry in the prior scope.

Nested modal scopes and spatial nearest-neighbor focus are intentionally outside
this contract.
