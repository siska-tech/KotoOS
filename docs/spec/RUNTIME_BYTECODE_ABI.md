# KotoRuntime Bytecode And Host ABI

This document fixes the first executable contract for `kotoruntime-bytecode`.
It covers the `KBC1` bytecode header, static verifier, host-call ABI, and the
per-frame execution model.

## Goals

- Keep VM state bounded and predictable on RP2040-class SRAM.
- Let bytecode live in a KPA asset, SD stream, or PSRAM-backed block cache.
- Make invalid bytecode and invalid host calls return deterministic errors.
- Keep the host API VM-neutral enough that a later Wasm path can reuse it.

## Bytecode Asset Layout

Koto bytecode v1 is little-endian. A bytecode asset starts with a fixed
64-byte `KbcHeader`, followed by code, optional read-only data, and optional
debug/name data.

```text
+--------------------------+ offset 0
| KbcHeader                |
+--------------------------+ header.code_offset
| KbcInsn[code_words]      |
+--------------------------+ header.rodata_offset
| read-only bytes          |
+--------------------------+ header.debug_offset
| optional debug bytes     |
+--------------------------+ header.bytecode_size
```

All offsets are relative to the start of the bytecode asset, not the start of
the `.kpa` file.

## Const heap image (rodata)

The `rodata` segment is the **initial image of the app heap** (KOTO-0139). Before
the entry word runs, the loader copies `rodata` into `heap[0..rodata_size]` and
leaves the rest of the heap zeroed — a single `memcpy`, no VM execution, no code-
window activity. This replaces baking const tables at startup with `store16`/
`store8` sequences (which ran as VM code and could stall the first frame).

Because `rodata` initializes the heap *prefix*, const data is addressed by the same
heap offsets as ordinary mutable buffers; the compiler simply places const-
initialized data within that prefix and emits its bytes here. The verifier requires
`rodata_size <= max_heap_bytes` so the copy is always in bounds. The loader reads
`rodata` from the asset (the simulator from the resident slice; PicoCalc straight
from the SD `.kbc`, since `rodata` is not staged into the PSRAM code window).

In the Koto app language, a top-level `data NAME = u16[...]` / `u8[...]` declaration
emits its little-endian bytes into `rodata`; `kbc-asm` exposes the same via a
`.rodata <hex>` directive.

## Debug Data

The optional debug section starts at `header.debug_offset` and uses the first
format below. It is generated by `kbc-asm` from `.debug_file` and `.loc`
directives; the Koto compiler emits those directives automatically.

```text
+--------------------------+ offset 0
| KdbgHeader               |
+--------------------------+
| File[files]              |
+--------------------------+
| LineEntry[entries]       |
+--------------------------+
```

`KdbgHeader` is little-endian:

| Offset | Size | Field       | Description                                  |
| :----- | :--- | :---------- | :------------------------------------------- |
| 0      | 4    | magic       | ASCII `KDBG`.                                |
| 4      | 2    | version     | Must be `1`.                                 |
| 6      | 2    | header_size | Must be `16`.                                |
| 8      | 2    | file_count  | Number of file table records.                |
| 10     | 2    | entry_count | Number of PC-to-source records.              |
| 12     | 4    | reserved    | Must be `0`.                                 |

Each file record is `path_len:u16` followed by UTF-8 path bytes. Each line entry
is 14 bytes:

| Offset | Size | Field      | Description                                      |
| :----- | :--- | :--------- | :----------------------------------------------- |
| 0      | 4    | pc         | Instruction index relative to `code_offset`.     |
| 4      | 2    | file_index | Index into the file table.                       |
| 6      | 2    | reserved   | Must be `0`.                                     |
| 8      | 4    | line       | 1-based source line.                             |
| 12     | 2    | column     | 1-based source column.                           |

Entries are sorted by `pc` and represent the source location active from that
instruction until the next entry. Missing debug data is valid; simulator
diagnostics then fall back to bytecode PC only.

## KbcHeader

`KbcHeader` is exactly 64 bytes.

| Offset | Size | Field           | Description                                                              |
| :----- | :--- | :-------------- | :----------------------------------------------------------------------- |
| 0      | 4    | magic           | ASCII `KBC1`.                                                            |
| 4      | 2    | version_major   | Must be `1`.                                                             |
| 6      | 2    | version_minor   | Must be `0` for the first prototype.                                     |
| 8      | 4    | header_size     | Must be `64`.                                                            |
| 12     | 4    | flags           | Reserved. Must be `0` in v1.                                             |
| 16     | 4    | bytecode_size   | Total bytecode asset size in bytes.                                      |
| 20     | 4    | code_offset     | Offset of the first instruction. Must be 4-byte aligned and at least 64. |
| 24     | 4    | code_size       | Code size in bytes. Must be a nonzero multiple of 4.                     |
| 28     | 4    | rodata_offset   | Offset of the const heap image, or `0` if absent. See "Const heap image". |
| 32     | 4    | rodata_size     | Const heap image size in bytes. Must be `0` when `rodata_offset` is `0`, and `<= max_heap_bytes`. |
| 36     | 4    | entry_word      | Entry instruction index, relative to `code_offset`.                      |
| 40     | 2    | max_stack_slots | Operand stack slots requested by the app.                                |
| 42     | 2    | max_call_depth  | Call stack frames requested by the app.                                  |
| 44     | 4    | max_heap_bytes  | App heap bytes requested by the app. `0` means no VM heap.               |
| 48     | 2    | host_abi_major  | Host ABI major version. Must match the runtime.                          |
| 50     | 2    | host_abi_minor  | Host ABI minor version requested by the app.                             |
| 52     | 4    | debug_offset    | Offset of optional debug data, or `0` if absent.                         |
| 56     | 4    | debug_size      | Debug data size in bytes. Must be `0` when `debug_offset` is `0`.        |
| 60     | 4    | reserved        | Must be `0`.                                                             |

The first runtime should cap `max_stack_slots`, `max_call_depth`, and
`max_heap_bytes` against package manifest memory limits and platform runtime
limits before allocating VM state.

## Instruction Encoding

The first VM uses fixed 32-bit instruction words:

```text
31          24 23          16 15                       0
+-------------+--------------+--------------------------+
| opcode: u8  | operand: u8  | immediate: u16           |
+-------------+--------------+--------------------------+
```

The initial instruction set should stay integer-first:

| Group        | Examples                                                                           | Notes                                                                                  |
| :----------- | :--------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------- |
| Control      | `nop`, `halt`, `br`, `br_if_zero`, `call`, `ret`                                   | Branch and call targets are instruction indices.                                       |
| Stack        | `push_i16`, `dup`, `drop`, `swap`                                                  | All stack movement is bounds-checked.                                                  |
| Locals       | `load_local`, `store_local`                                                        | Locals live in the current call frame.                                                 |
| Integer math | `add_i32`, `sub_i32`, `mul_i32`, `div_i32`, `and_i32`, `or_i32`, `xor_i32`, shifts | Division by zero traps with a VM error.                                                |
| Memory       | `load8`, `store8`, `load16`, `store16`, `load32`, `store32`                        | Accesses are relative to the app heap or host-validated buffers.                       |
| Host         | `host_call`                                                                        | `operand` selects the host-call ID; `immediate` is reserved unless a call declares it. |

The first assembler may generate only the subset needed by fixtures. The
verifier still treats unknown opcodes as fatal.

## Verifier

Bytecode must be verified before the runtime starts an app. Verification is a
single bounded pass over the header and code region, plus range checks for data
regions.

The verifier rejects:

- Bad magic, unsupported version, wrong header size, nonzero reserved fields, or
  nonzero v1 flags.
- Any offset/size pair outside `bytecode_size`, any integer overflow while
  computing ranges, or overlapping ranges that are not explicitly allowed.
- `code_size == 0`, non-4-byte-aligned code, or `entry_word` outside the code
  word count.
- Resource requests above the runtime or package memory limits.
- Unknown opcodes.
- Branch or call targets outside the code word count.
- `ret` from the entry frame when the instruction is not treated as app exit.
- Static stack underflow for straight-line instructions.
- Host-call IDs unknown to the selected host ABI version.
- Memory op widths or declared buffer indexes that cannot be proven valid by
  the instruction encoding.

The first verifier does not need full control-flow stack-depth proof. It may
combine simple static checks with runtime stack checks. Runtime checks remain
mandatory even after verification.

## Runtime Errors

The VM returns errors; it must not panic on malformed app input.

| Error                              | Meaning                                              |
| :--------------------------------- | :--------------------------------------------------- |
| `BadHeader`                        | Header, version, offset, or size validation failed.  |
| `BadInstruction`                   | Unknown opcode or malformed operand.                 |
| `BadBranch`                        | Branch/call target is outside code.                  |
| `StackUnderflow` / `StackOverflow` | Operand stack bounds failure.                        |
| `CallDepthExceeded`                | Call stack limit exceeded.                           |
| `MemoryOutOfBounds`                | App heap or buffer access is outside bounds.         |
| `HostCallDenied`                   | Host call is unknown, unsupported, or not permitted. |
| `HostCallFailed`                   | Host service returned a normal recoverable failure.  |
| `FuelExhausted`                    | Frame or run budget was consumed.                    |
| `Exited`                           | App requested exit or returned from entry.           |

## Host Call ABI

The bytecode `host_call` instruction uses a numeric host-call ID. Arguments and
return values are `i32` stack values unless a call explicitly uses an app memory
buffer. Arguments are popped right-to-left and return values are pushed in the
listed order.

Every host call receives an implicit `AppContext` from the runtime:

- `app_id`
- package identity
- sandbox root identity
- package asset reader
- save-data mount or namespace
- current frame input snapshot
- platform capability flags

Bytecode cannot forge or pass `AppContext`. File calls must use this implicit
context for sandbox resolution.

### Return Convention

Host calls return either success or failure without panicking. **Every call pushes
a fixed number of values — its documented result slots, then a status word —
regardless of success or failure**, so static stack accounting matches the runtime
on both paths:

- Success pushes the documented result values and then `0`.
- Failure pushes the same number of result slots, each `0`, and then a negative
  status equal to `-error_code` (e.g. `-3` for `PermissionDenied`).
- Calls with no result push only the status word: `0` for success, `-error_code`
  for failure.

The top stack value after a host call is always the status (`0` ok, `< 0` error).
App code may ignore it, but a host call that can fail must be treated as pushing
its declared result count plus the status on every path — dropping only the
success shape leaks the operand stack on failure.

Host-call IDs `0x21` and `0x60`–`0x66` were added at host ABI minor version `1`.
They form a VM-neutral text-composition / text-buffer service: the host owns the
IME and editor models, and bytecode drives composition, conversion, cursor
movement, deletion, loading, and read-back across the boundary. The romaji→kana
and dictionary-conversion algorithms stay in host code so the same calls can back
a later Wasm runtime. The static verifier models each host call's
`(args_popped, values_pushed)` effect (success shape) so looping interactive apps
verify accurately; runtime stack-bound checks remain authoritative.

Host ABI minor version `2` adds the `ime_toggle` input intent and `IME_TOGGLE`
key kind. It does not add a new host-call ID.

Host ABI minor version `3` adds editor viewport query calls for rendering visible
memo lines, cursor position, and scroll indicators without copying the whole
document into app code every frame.

Host ABI minor version `4` adds `edit_view_metrics` so bytecode apps can align
text rows and caret rectangles to the host-selected bitmap font instead of
duplicating simulator font constants.

Host ABI minor version `5` adds `edit_cursor_status` for apps that need a
human-readable cursor position but do not yet have integer-to-string formatting.

Host ABI minor version `6` adds `edit_total_lines` for deterministic scrollbar
track/thumb rendering without copying the whole document into app code.

Host ABI minor version `7` adds `dir_list` so an app can enumerate its own
save-data sandbox directory (e.g. for a file open/save dialog) without the host
exposing filesystem paths, plus the `open` input intent (bit 16) for surfacing
such a dialog.

Host ABI minor version `8` wires audio output (KOTO-0095): the `audio_submit_i16`
low-level PCM primitive (id `0x30`, reserved since the first prototype) is now
implemented by the simulator, and the host-owned audio service adds `play_sfx`
(`0x31`), the now-legacy `play_bgm` (`0x32`), and `stop_bgm` (`0x33`). The synth
runs host-side so no PCM lives in the VM heap. Sound-effect ids remain an
`audio_id` convention; app-specific music moved to package assets in ABI minor 10.

Host ABI minor version `9` adds `edit_reserve_rows` (id `0x71`, KOTO-0099) so an
app can tell the host how many bottom viewport rows an overlay (such as the IME
conversion panel) covers. The host keeps the cursor scrolled above the reserved
band, so the line being edited is never hidden under the panel.

Host ABI minor version `10` adds `play_bgm_asset` (id `0x34`). App-specific
KotoMML is shipped in the package and can be changed by rebuilding only the app
and its assets; it is no longer compiled into the host music-id bank.

Host ABI minor version `11` adds `play_sfx_asset` (id `0x35`). Game-specific
effects now ship with each app; only system-shell cues remain host-owned.

Host ABI minor version `12` adds `edit_configure` (id `0x72`) so editor geometry
is selected by the app instead of being hard-coded for Memo in the host.

Host ABI minor version `13` adds the Game2D retained tile renderer (KOTO-0135):
`game2d_set_tile` (id `0x14`), `game2d_clear_layer` (id `0x15`), and
`game2d_present` (id `0x16`). The host holds a 16x16-tile layer; an app writes
only the cells that change and presents once per frame, so per-frame draw work is
proportional to what changed rather than to board fill. Tile art stays in the app
heap, referenced by `tile_ref` byte offset (reusing the `draw_pixels_rgb565` heap
re-read). IDs `0x17`–`0x1F` stay reserved for the follow-up sprite/tile-define
calls (see [GAME2D_ABI.md](GAME2D_ABI.md)).

### Input Snapshot Fields

The frame-stable input snapshot carries, in addition to the held/pressed button
bits returned by `input_snapshot`:

- `text_codepoint` — the Unicode scalar typed this frame, or `0` when none.
- `intent_bits` — edit-intent flags returned by `text_input`. The host maps
  physical keys to intents; bytecode reads them to choose which host calls to make.

| Bit | Intent       | Bit | Intent       |
| :-- | :----------- | :-- | :----------- |
| 0   | `shift`      | 8   | `up`         |
| 1   | `convert`    | 9   | `down`       |
| 2   | `commit`     | 10  | `home`       |
| 3   | `cancel`     | 11  | `end`        |
| 4   | `backspace`  | 12  | `newline`    |
| 5   | `delete`     | 13  | `save`       |
| 6   | `left`       | 14  | `exit`       |
| 7   | `right`      | 15  | `ime_toggle` |
|     |              | 16  | `open`       |

`ime_feed_key` `kind` values: `0` character (uses `codepoint`), `1` shift, `2`
convert, `3` commit, `4` cancel, `5` other, `6` toggle IME enabled state.
`edit_move` `dir` values: `0` left, `1` right, `2` up, `3` down, `4` home,
`5` end. `edit_delete` `kind` values: `0` backspace, `1` forward delete.

## One-Frame Execution Model

KotoRuntime runs inside the platform frame loop. It is cooperative and bounded:
an app receives a per-frame instruction budget and must yield, exit, block on a
host service, or be stopped when fuel is exhausted.

```text
poll input
begin frame
  shell or active app receives the stable input snapshot
  runtime executes up to frame_fuel instructions
  host draw calls append dirty rectangles or scanline work
  audio host calls fill or queue bounded audio buffers
end frame
flush dirty drawing through HAL
service audio/storage continuations
```

### Frame Rules

- Input is sampled once before VM execution and remains stable for all
  `input_snapshot` calls in that frame.
- Draw calls do not push pixels directly to hardware. They record render work or
  update a bounded surface abstraction, then the HAL flushes dirty regions.
- File and asset calls must be nonblocking from the VM perspective. If the
  backend cannot finish within the call budget, it returns `WouldBlock`; the app
  can retry on a later frame.
- Audio calls enqueue bounded buffers and return the number of frames accepted.
- `yield_frame` returns control to the shell/platform loop without destroying VM
  state.
- `exit` or returning from the entry frame transitions the app to an exited
  state and releases app-owned handles.
- `FuelExhausted` ends only the current frame by default. Repeated exhaustion
  without yielding may be promoted to an app error by policy.

### Suggested Initial Budgets

These are the upper `RuntimeLimits` *caps* a verified app may request, not the
fixed sizes the simulator actually instantiates (those are the smaller
[Simulator VM Profile](#simulator-vm-profile) below):

| Budget        | Cap                       | Reason                                                               |
| :------------ | :------------------------ | :------------------------------------------------------------------- |
| Operand stack | 256 slots                 | 1 KB for 32-bit values.                                              |
| Call depth    | 32 frames                 | Enough for small apps while catching recursion.                      |
| App heap      | 0-16 KB                   | Start at zero for draw/input fixtures; add heap for text/file tests. |
| Frame fuel    | 2,000-10,000 instructions | Bounded enough for interactive shell testing.                        |
| Open handles  | 8                         | Keeps file/asset state small and auditable.                          |

Device builds should derive actual caps from the package manifest memory field
and platform memory profile.

### Simulator VM Profile

The simulator instantiates one fixed VM shape that every committed app must load
on, and the Koto compiler floors each app's header request to match it. These are
deliberate, predictable bounds for RP2040-class SRAM — small by default and raised
only when a real app needs it, with the rationale recorded — not hard limits of
the bytecode itself.

| Capacity         | Value      | Constant (sim / compiler)             | Per-VM cost | Notes |
| :--------------- | :--------- | :------------------------------------ | :---------- | :---- |
| Operand stack    | 16 slots   | `SIM_VM_STACK_SLOTS` / `MIN_STACK`    | 64 B        | Depth of expression/argument evaluation. |
| Local file       | 48 slots   | `VM_LOCAL_SLOTS`                      | 192 B       | 45 user + 3 codegen scratch; see below. |
| Call depth       | 4 frames   | `SIM_VM_CALL_DEPTH` / `CALL_PROFILE`  | 16 B        | Non-recursive; flat locals across frames. |
| App heap         | per-app, ≤16 KB | `RuntimeLimits::max_heap_bytes` (ceiling) | header-sized | Buffers + materialized string/byte data. Each app gets a heap sized to its own KBC header request (per-app profile, KOTO-0096); 16 KB is the device ceiling. |
| Frame fuel       | 60000 insn | `SIM_FRAME_FUEL` (`frame_fuel`)       | —           | Host-side per-frame instruction budget; not a header request. Sized for a full-screen tile/sprite app (KotoBlocks). |

Stack slots, call depth, and frame fuel come from one profile,
`RuntimeLimits::simulator_default()` (KOTO-0060): the simulator derives its `SIM_VM_*`
/ `SIM_FRAME_FUEL` constants and the compiler derives its `MIN_STACK` / `CALL_PROFILE`
request floors from it, with the invariant `MIN_STACK == SIM_VM_STACK_SLOTS`. The
**heap is per-app** (KOTO-0096): the compiler writes each app's actual need to the
header, the simulator gives the VM a heap of exactly that size, and `verify_kbc`
rejects any header above the `max_heap_bytes` ceiling — so verification is never more
permissive than launch. The compiler reserves three scratch local slots (a
return-value slot and two `%`-lowering operand slots) within the
`koto_core::runtime::VM_LOCAL_SLOTS` file; they **float** to sit just above each
program's user-slot high-water mark rather than at fixed top-of-file indices
(KOTO-0146), so the VM's `local_slots_peak` (the highest slot *index* touched)
tracks real local pressure instead of always reaching 48 for any app that uses a
return value or `%`. This is a compiler slot-numbering choice only: the user-slot
cap (45), the local-file size (48), and the bytecode ABI are unchanged.

**The local file is not part of the header.** Unlike stack/call/heap requests, the
VM's local register file (`locals: [i32; VM_LOCAL_SLOTS]`) is a fixed VM capacity,
not declared per app. It is **shared across all of an app's functions** — the
compiler assigns each function a non-overlapping `slot_base` — and `store_local` /
`load_local` index it directly (operand byte, so up to 256 slots are encodable).
`let`s are block-scoped and their slots are reused once a block ends (KOTO-0092), so
the ceiling is the *peak* number of simultaneously-live locals across the call chain
(less the three scratch slots), not a program's static total of `let`s.

Raising any of these is a deliberate change: state the per-VM byte cost, keep the
paired invariants in sync, and prefer the structural fix (e.g. local slot reuse)
over repeated bumps. The memo app is the usual forcing function — it grows against
these bounds first.

## Validation Fixtures

The first VM tests should include:

- Header rejection for bad magic, unsupported version, bad offsets, and excessive
  resource requests.
- Verifier rejection for unknown opcode, bad branch target, static stack
  underflow, and unknown host call.
- A draw/input/exit fixture that runs for one frame and exits cleanly.
- A file fixture that proves virtual paths use the app sandbox identity.
- A fuel fixture that yields `FuelExhausted` without corrupting VM state.
