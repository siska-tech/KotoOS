# KotoVN Script And Image Pipeline Spike

This spike sketches the smallest KotoVN pipeline that can feed later runtime and
engine work without deciding the full KotoRuntime implementation.

## Tiny Script Fixture

The first fixture is [intro.kvn](../../harness/fixtures/kotovn/intro.kvn). It uses
one command per line and keeps command arguments package-relative:

```text
background assets/bg_room.rle
text speaker="Koto" "Welcome to the tiny VN fixture."
text "Backgrounds are streamed, decoded, and drawn by the host engine."
end
```

The initial command set is intentionally small:

| Command | Arguments | Meaning |
| :------ | :-------- | :------ |
| `background` | `asset_path` | Load or select an indexed-color RLE background asset. |
| `text` | optional `speaker="..."`, quoted UTF-8 body | Show one textbox of dialogue or narration. |
| `end` | none | Stop the script and return to the launcher or caller. |

Parsing rules for the later prototype:

- Lines are UTF-8 text and may end with LF or CRLF.
- Empty lines and lines starting with `#` are ignored.
- Asset paths use the KPA package namespace and the same path restrictions as
  other package assets.
- The first parser should reject unknown commands, malformed quoted strings, and
  missing assets instead of falling through silently.

## Image Asset Inputs

KotoVN backgrounds should start as indexed-color RLE assets so the host can keep
the VM small and perform bounded decode work. The expected source-side metadata
is captured in [image_inputs.json](../../harness/fixtures/kotovn/image_inputs.json).

Minimum input contract:

| Field | Initial value | Notes |
| :---- | :------------ | :---- |
| `format` | `kotovn-image-inputs` | Source-side metadata format. |
| `background.size` | `320x200` | Matches the KotoDOS-style game region for now. |
| `background.palette` | 256 RGB888 entries | Packer converts colors to RGB565 or a device palette table. |
| `background.encoding` | `rle-index8` | Runs are `(count:u8, index:u8)` pairs in row-major order. |
| `background.transparent_index` | `null` for backgrounds | Sprites may define transparency later. |
| `background.package_type` | `image-rle-index8` | Manifest asset type used by fixtures and packer reports. |

The decoder should be able to stream the RLE payload from SD or PSRAM block
storage, expand only the requested scanlines or dirty rectangle, and write RGB565
pixels through the host render path. Full-screen RGB565 expansion is a host
cache decision, not a VM requirement.

## VM And Host Boundary

The visual novel script belongs to app data, but the first KotoVN engine should
keep heavy media handling in host engine code.

| Piece | Runs In | Reason |
| :---- | :------ | :----- |
| Script bytecode or parsed command cursor | VM | App logic decides which line, branch, or scene is active. |
| Save-state variables and choice decisions | VM | These are app-owned state and should stay sandboxed. |
| Command validation against package asset names | Host engine | The host already owns package metadata and asset handles. |
| RLE decode and palette conversion | Host engine | Keeps VM heap small and allows bounded scanline/rect decode. |
| Background dirty-rect scheduling | Host engine | Integrates with `RenderSurface`, HAL flushing, and PSRAM policy. |
| Text shaping/layout call | Host engine | Reuses core text/font paths and avoids duplicating glyph code in VM. |

For the first prototype, the VM can issue host calls such as
`kotovn_background(asset_id)` and `kotovn_text(speaker, body)`. The host engine
resolves asset IDs, streams media, records dirty rectangles, and returns normal
host-call errors for missing assets or unsupported encodings.

## Open Questions

- Whether `.kvn` remains a text asset or is compiled into compact bytecode.
- Whether scene-local asset IDs should be generated by the packer.
- How choices, jumps, and save slots map onto the general KotoRuntime host ABI.
