# KOTO-0048: App Build And Package Loop

- Status: done
- Type: harness
- Priority: P0
- Requirements: FR-PKG-1, FR-PKG-2, FR-PKG-3, FR-SIM-3, FR-SIM-5, NFR-DEV-3, NFR-DEV-4
- Prerequisites: KOTO-0044, KOTO-0046

## Goal

Create a reproducible host-side build loop that compiles app source, emits
`KBC1`, updates the mock SD card package layout, and verifies that committed
fixtures match their sources.

## Acceptance Criteria

- [x] A single documented command rebuilds source apps into
      `sdcard_mock/bytecode/*.kbc`.
- [x] App sources live in a stable tree such as `apps/<app_id>/src`.
- [x] The project harness verifies that generated bytecode matches committed
      bytecode and passes `verify_kbc`.
- [x] Manifest entries are checked against generated bytecode paths and package
      asset declarations.
- [x] The build loop can be used by Koto Memo without hand-editing binary blobs.

## Notes

This issue turns the compiler into a practical app workflow. It should integrate
with the existing local checks instead of creating a separate manual ritual.

Implemented as `apps/apps.json` (the app registry) plus `harness/build_apps.py`:
`python harness/build_apps.py` rebuilds each registered app's committed
`sdcard_mock` bytecode, and `--check` (wired into `harness/check_all.py` as the
"App build sync" step) fails on source/bytecode drift or a manifest entry that
does not point at the built output. Each registry entry has a `kind` — `koto`
(high-level source via `koto-compiler`, which runs `verify_kbc`) or `asm`
(low-level IR via `kbc-asm`) — so Koto Memo can switch from the assembly fixture
to `apps/memo/src/main.koto` in KOTO-0041 by changing its registry entry only.
